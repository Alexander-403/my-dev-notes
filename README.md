## ООП

### Класс
Это некое описание характеристик приводя пример это чертеж в контексте ооп эти характеристики являются свойствами а действия которые может совершать тот или иной объект называются методами

### Объект
Это конкретный представитель класса то есть если приводить пример то класс это чертеж а объект это готовый проект  
Подводя итог классов и объектов то класс это описание характеристик а объект это конкретный экземпляр у которого каждая характеристика имеет значение

### Инкапсуляция
Суть понятия инкапсуляция заключается в том что сам класс по сути является капсулой которая содержит в себе свойства и методы позволяя их тем самым объединить

### Сокрытие
Это механизм который ограничивает доступ к свойствам и методам для этого есть два модификатора доступа private и public свойства и методы помеченные модификатором private могут использоваться только внутри класса

### Наследование
Это механизм позволяющий создавать новые классы на основе существующих проще говоря при наследовании исходный класс (класс-родитель) расширяется дополнительной информацией

### Полиморфизм
Можно воспринимать как некий принцип который позволяет одному и тому же фрагменту кода работать с разными типами данных    

Полиморфизм делиться на два типа это  
параметрический(Истинный) и ad-hoc(Мнимый)  
Ad-hoc это особый вид полиморфизма, где функция ведет себя по-разному в зависимости от типов аргументов  
Самым простым примером будет данный кусок кода где в одном случае получается конкантенация строк а в другом обычное сложение:

### Параметрический
Это когда одна и та же функция работает с любыми типами данных то есть один код для всех типов данных

### Агрегация и композиция
Агрегация это вид связи между классами при которой один объект (целое) включает в себя другие объекты (части), причем части могут существовать независимо от целого простыми словами агрегация это "Часть МОЖЕТ жить без целого" (слабая связь)

Композиция это принцип организации кода, при котором один класс содержит в качестве полей объекты других классов, создавая отношение «часть-целое» простыми словами  
Композиция это "Часть НЕ МОЖЕТ жить без целого" (сильная связь)

### Абстракция
это принцип, который позволяет скрыть сложную реализацию и показать пользователю (программисту) только существенные характеристики и функциональность объекта.

Простая аналогия: Вы пользуетесь автомобилем. Чтобы ехать, вам не нужно знать, как именно работает система впрыска топлива или электронный блок управления. Вы взаимодействуете с объектом "Автомобиль" через интерфейс: руль, педали газа и тормоза, рычаг коробки передач. Вся сложность "спрятана" под капотом. Это и есть абстракция.

## Хранение в базах данных

База данных - это организованная коллекция структурированной информации или данных, обычно хранящихся в электронном виде в компьютерной системе. Структура БД спроектирована так, чтобы обеспечивать эффективное хранение, управление, извлечение и модификацию данных

### Виды баз данных

**Реляционные**  
Данные организованы в виде таблиц, каждая из которых представляет определённое отношение. Таблицы состоят из строк (записей) и столбцов (атрибутов). Строки содержат информацию о конкретных объектах, а столбцы описывают их свойства.  
Особенности:  
* Таблицы связаны между собой через первичные и внешние ключи, чтобы объединять данные из нескольких таблиц.  
* Для управления реляционными базами используется язык запросов SQL.  
Примеры: MySQL, PostgreSQL, Oracle Database, Microsoft SQL Server.

**Документоориентированные**  
Данные хранятся в виде документов, обычно в форматах JSON или XML. Документы могут быть иерархическими или плоскими и представляют собой независимые единицы информации.  
Особенности:  
* Документы могут быть организованы в коллекции.  
* Документы адресуются с помощью уникального ключа, обычно это строка, которая генерируется автоматически.  
* Помимо простого поиска документов по ключу, документоориентированные базы данных предоставляют свой язык запросов.  
Примеры: MongoDB, CouchDB.

**Графовые**  
Особенность — применение структуры графа для хранения данных и запросов. В графовой БД два базовых компонента:  
* Узлы — экземпляры, или сущности данных.  
* Ребра — компонент, отображающий связь между узлами.  
Особенности:  
* Узлы со свойствами взаимосвязаны через рёбра, причём связь может быть ориентированной и неориентированной: в ориентированных графах у каждого ребра есть чёткое направление от точки А к точке В, в неориентированных — нет.  
* Графовые БД лучше справляются со сложными запросами, подразумевающими анализ связей.  
Примеры: Neo4j, ArangoDB, TigerGraph.

### Виды связей в базах данных

**Один-к-одному (1:1)**  
Каждой записи в одной таблице соответствует ровно одна запись в другой. Это редкий случай, но используется, когда необходимо разгрузить таблицу или выделить чувствительные данные.  
Пример: таблица Users и таблица Passports, где каждому пользователю соответствует один паспорт. Часто связь 1:1 реализуется при проектировании сложных систем, где нужно отделить редко используемые поля или сведения с ограниченным доступом

**Один-ко-многим (1:N)**  
Каждая запись в первой таблице может быть связана с множеством записей во второй, но обратное — нет. Это самая распространённая форма связи.  
Пример: один автор (таблица Authors) может написать много книг (таблица Books), но каждая книга принадлежит только одному автору. Этот тип связи также актуален для категорий товаров, клиентов и их заказов, сотрудников и их задач.

**Многие-ко-многим (M:N)**  
Каждая запись из одной таблицы может быть связана с несколькими записями другой таблицы и наоборот. Для реализации такой связи используется промежуточная таблица.  
Пример: студенты и курсы. Один студент может посещать несколько курсов, и каждый курс — быть доступным для разных студентов. Промежуточная таблица (StudentCourse) будет хранить пары соответствий.  
Важно: реализация связи «многие-ко-многим» обычно предполагает создание третьей, промежуточной таблицы, которая служит для хранения связей между записями в двух связанных таблицах.

### СУБД (Система управления базами данных)
СУБД - это программное обеспечение, с помощью которого можно создавать, хранить, изменять и анализировать данные. По сути, СУБД — это посредник между пользователем и физическим хранилищем данных, который автоматизирует множество операций и обеспечивает целостность информации. Работа СУБД можно представить как многоуровневую архитектуру, где каждый уровень выполняет свои специфические функции. Эта архитектура обеспечивает разделение между логическим представлением данных (тем, что видят пользователи) и физическим хранением информации (как данные фактически располагаются на дисках).

Функции  
* Управление доступом — возможность настроить, кто и какие действия может выполнять: кто имеет право только читать данные, а кто — вносить изменения или удалять записи.  
* Резервное копирование и восстановление — благодаря встроенным средствам резервного копирования СУБД может сохранять «снимки» состояния базы и восстанавливать её после сбоев — полностью или частично.  
* Оптимизация производительности — системы анализируют нагрузку, подсказывают, где узкие места, и автоматически оптимизируют выполнение запросов.  
* Поддержка языков запросов и API — СУБД обеспечивает доступ к данным через стандартизированные языки, такие как SQL, а также через программные интерфейсы для подключения внешних приложений.  
* Обеспечение целостности данных — система следит за тем, чтобы данные не противоречили заданным правилам.  
* Многопользовательский режим — СУБД позволяет нескольким пользователям одновременно работать с базой данных, не мешая друг другу

### SQL
SQL (Structured Query Language) -не самостоятельный язык программирования, а язык структурированных запросов. Он предназначен для работы с реляционными базами данных (БД) - массивами информации, которые связаны между собой и представлены в виде таблиц. (Стоит изучать подробнее)

### NoSQL
NoSQL - обозначение нереляционных систем управления базами данных (СУБД). Они отказываются от традиционной табличной структуры реляционных баз в пользу альтернативных моделей хранения.

типы NoSql (нереляционных баз данных)  
* Документоориентированные — хранят данные в полуструктурированных документах (JSON, BSON, XML).  
* Хранилища «ключ-значение» — простейшая модель, где каждое значение доступно по уникальному ключу.  
* Колоночные — оптимизированы для аналитических запросов над большими наборами данных.  
* Графовые — специализируются на хранении сложных взаимосвязей между объектами.  
Особенности:  
* Строго определённой схемы взаимосвязи между данными нет — информационная модель определяется СУБД «по ходу дела» в процессе работы приложения.  
* В нереляционных базах данных не применяется именно SQL, но разработчики многих NoSQL-СУБД применяют языки управления данными, в той или иной мере похожие на него по синтаксису.  
* В нереляционных базах данных реализована горизонтальная масштабируемость — за счёт соединения нескольких независимых друг от друга серверов, каждый из которых обрабатывает только часть данных, а не весь массив.  
        Примеры: MongoDB, Redis, Apache Cassandra  
Принцип работы  
В отличие от реляционных баз данных, где данные должны соответствовать строго определённой схеме, NoSQL-базы позволяют хранить их в различных форматах без необходимости заранее определять структуру. Это особенно полезно в ситуациях, когда структура данных часто меняется или когда данные имеют сложную вложенную структуру

### Требование ACID
ACID - набор требований к транзакционной системе, обеспечивающий наиболее надёжную и предсказуемую её работу.  
Некоторые требования ACID:  
1. Атомарность. Гарантирует, что никакая транзакция не будет зафиксирована в системе частично. Будут либо выполнены все её подоперации, либо не выполнено ни одной.  
2. Согласованность. Транзакция, достигающая своего нормального завершения и тем самым фиксирующая свои результаты, сохраняет согласованность базы данных. Другими словами, каждая успешная транзакция по определению фиксирует только допустимые результаты.  
3. Изоляция. Во время выполнения транзакции параллельные транзакции не должны оказывать влияния на её результат.  
4. Устойчивость. Независимо от проблем на нижних уровнях (к примеру, обесточивание системы или сбои в оборудовании) изменения, сделанные успешно завершённой транзакцией, должны остаться сохранёнными после возвращения системы в работу.

## Работа с сетью

Сеть - любая тех система предназначенная для передачи по линиям связи информации доступ к которой осуществляется с помощью средств вычислительной техники(по простому сеть это два и более устройств взаимодействующих между собой  через какую либо среду для передачи данных; беспровод, оптический канал, кабель)

Osi - которая предназначена для передачи данных разделенная на семь частей (семиуровневая модель)  
1физический уровень -   
   • определение физических характеристик интерфейсов и среды передачи;  
* установление, поддержание и разрыв физических соединений;  
* преобразование цифровых данных в сигналы, подходящие для передачи;  
* синхронизация передачи битов между устройствами;  
* определение топологии сети и способа подключения устройств.  
«уровень кабеля(напрежение радио волны и тп)» к устройствам физического уровня относятся хабы, повторители, трансиверы, сетевые адаптеры и физические интерфейсы маршрутизаторов и коммутаторов

2канальный уровень - Принимает поток битов с физического уровня и организует их в логические структуры — кадры (frames).  
 
* Кадрирование — разбиение непрерывного потока битов на управляемые блоки данных с добавлением заголовков и концевиков для идентификации начала и конца кадра.  
* Физическая адресация — добавление MAC-адресов отправителя и получателя в заголовок кадра для идентификации устройств в локальной сети.  
* Контроль потока — регулирование скорости передачи данных для предотвращения перегрузки приёмного устройства.  
* Обнаружение и исправление ошибок — выявление повреждённых кадров и организация их повторной передачи при необходимости.

3сетевой уровень Отвечает за маршрутизацию пакетов данных между различными сетями. Некоторые функции:  
* Логическая адресация — присвоение и управление IP-адресами.  
* Маршрутизация — определение оптимального пути для передачи пакетов данных от отправителя к получателю через множество промежуточных узлов.  
* Фрагментация и сборка пакетов — разбиение больших пакетов на меньшие фрагменты перед отправкой и сбор их обратно на стороне получателя.  
* Контроль перегрузок — реализация механизмов для предотвращения перегрузок в сети.

4транспортный уровень - обеспечивает передачу данных по сети  
Tcp - используется когда трафик уязвим к потерям(документ по почте)  
udp - когда не критичны потери фризы и тп 

5сеансовый уровень. Обеспечивает синхронизацию между устройствами и восстанавливает связь, если она прервалась. Главная задача этого соединения — обеспечить стабильное соединение для передачи данных. Примером могут служить: видиоконфиренции звонки и тп

6уровень представления Здесь данные должны преобразоваться в унифицированный формат, чтобы их можно было передавать на разные устройства и операционные системы. (При отправке изоображения оно изначально приходит в виде набора 0 и 1 как раз таки на 6 уровн7е оно преобразуется в изображение)

7уровень приложения(прикладной) Верхний уровень модели, обеспечивающий взаимодействие пользовательских приложений с сетью. Позволяет приложениям использовать сетевые службы: удалённый доступ к файлам и базам данных, пересылка электронной почты.  

Протоколы  
http Это протокол прикладного уровня, разработанный для обмена данными между клиентом (например, веб-браузером) и сервером в сети Интернет.  
HTTP работает по принципу «запрос-ответ»:  
1. Клиент отправляет запрос на сервер, сообщая, что ему нужно (например, страница index.html или изображение logo.png).  
2. Сервер ищет нужный ресурс на своих дисках и, если находит, отправляет ответ обратно браузеру.  
3. Браузер получает ответ и отображает его — пользователь видит загруженную страницу или изображение.

https протокол защищённой передачи данных, расширение протокола HTTP. Его задача — защитить информацию, которую отправляют и получают на сайте: логины, пароли, данные банковской карты.Когда пользователь заходит на сайт с протоколом HTTPS, браузер и сервер устанавливают защищённое соединение друг с другом. Это похоже на секретный канал, по которому передаются все данные: посторонние не могут подслушать или вмешаться.  
1. Установка соединения — клиент (браузер) инициирует запрос на создание HTTPS-соединения с сервером.  
2. Обмен сертификатами — сервер отправляет свой цифровой сертификат клиенту.  
3. Проверка сертификата — клиент проверяет сертификат и, если он действителен, генерирует симметричный ключ для шифрования данных.  
4. Шифрование данных — все передаваемые данные между клиентом и сервером шифруются с использованием этого ключа.  
5. Завершение соединения — после обмена данными соединение может быть закрыто, или клиент и сервер могут оставить его открытым для дальнейших запросов.

Mac адрес  
Простыми словами Mac адрес это адрес(индикатор) устройства присваемый производителем по этому адресу можно почти конкректно понять что это за устройство  
MAC-адрес - аппаратный идентификатор сетевого интерфейса, присваиваемый производителем. Представлен в 48-битном формате и записывается шестнадцатеричными парами, например: 1:B2:C3:D4:E5:F6.  

MAC-адрес состоит из двух сегментов:  
1. Первые 3 байта (OUI, Organizationally Unique Identifier) — код производителя устройства. Например, у Intel он может начинаться с 00:1B:44.  
2. Оставшиеся 3 байта — уникальный серийный номер конкретного устройства, назначенный изготовителем

Dns сервер  
DNS - система доменных имён, которая позволяет сопоставлять домены сайтов с IP-адресами, необходимыми для связи между устройствами в сети  
Процесс получения IP-адреса для имени состоит из ряда шагов:  
 Как работает  
1. Проверка на локальной машине — система смотрит файл hosts. Если запись найдена, браузер сразу подключается к сайту. Если нет, формируется рекурсивный запрос к DNS-серверу провайдера.  
2. Запрос к DNS-серверу — сервер проверяет собственный кеш — временное хранилище свежих ответов. Если там уже есть IP для имени, он сразу отправляется в браузер. Иначе начинается поиск через корневые серверы.  
3. Корневые серверы — не возвращают IP конкретных сайтов, но знают адреса TLD-серверов и перенаправляют к ним запросы.  
4. TLD-серверы — обслуживают домены верхнего уровня — «.com», «.org», «.ru» и другие. Хранят адреса авторитативных серверов доменов второго уровня и подсказывают резолверу следующий шаг.  
5. Авторитативные серверы — последнее звено в цепочке. Содержат ресурсные записи конкретной зоны и возвращают окончательный ответ — IP-адрес искомого имени.

Сетевые порты  
Сетевой порт — это виртуальная или физическая точка взаимодействия между устройствами в компьютерной сети, через которую передаются данные. Каждый порт связан с определённым приложением или сервисом, работающим на устройстве, и определяется номером.Простыми словами сетевые порты предназначены для того что бы система понимала какому процессу предназначены данные

Маски и подсети  
Подсеть - это логически выделенный сегмент общей сети. Она позволяет логически разделять сеть на меньшие сегменты, чтобы уменьшить трафик и повысить безопасность.  
Маска подсети — это 32-битное число, служащее битовой маской для разделения сетевой части (адреса подсети) и части хоста IP-адреса. Она представляет собой последовательность битов, где единицы обозначают адрес сети, а нули — адрес узла.

Firewall  
Firewall - программное или аппаратное средство, которое защищает компьютерные сети и устройства от несанкционированного доступа. Основная задача файрвола — фильтрация входящего и исходящего сетевого трафика. Он устанавливает правила для управления этим трафиком, блокируя подозрительные соединения и разрешая безопасные.  

Типы файрволов:  
* Аппаратные — физические устройства, которые устанавливаются между локальной сетью и интернетом.  
* Программные — приложения, которые устанавливаются на компьютеры или серверы.  
* Сетевые — работают на уровне сети и фильтруют трафик между различными сегментами сети.  
* Межсетевые экраны следующего поколения (NGFW — next-generation firewall) — предлагают расширенные функции, такие как инспекция SSL, обнаружение угроз и интеграция с другими системами безопасности.

API  
API (Application Programming Interface) — это программный интерфейс, который позволяет разным программам взаимодействовать друг с другом — передавать и получать данные по запросу. API выполняет роль посредника и делает возможным совместную работу программ.  

Работа API строится на обмене информацией между двумя сторонами: клиентом и сервером:  

1. Запрос — клиент отправляет запрос серверу через API. Запрос обычно состоит из адреса (URI), команды («получить данные», «добавить запись» и т. д.) и иногда дополнительной информации в виде заголовков или данных.  
2. Обработка запроса — получив запрос, API отправляет его на сервер, который проверяет, какую именно информацию или действие запрашивают, и выполняет необходимые операции.  
3. Ответ от сервера — после обработки сервер отправляет ответ обратно API. Ответ содержит либо запрошенные данные, либо результат выполненного действия, либо сообщение об ошибке.  
4. Доставка ответа — API передаёт полученные данные клиенту.

## Многопоточность

Многопоточность (или multithreading) — это способность системы выполнять несколько потоков (нитей выполнения) одновременно в рамках одного процесса. При этом каждый поток — это отдельная последовательность команд, которая может выполняться независимо от других, что позволяет эффективно использовать ресурсы процессора и ускорять выполнение задач.

• Процесс/Поток

Процесс — это экземпляр программы. Каждый запущенный процесс в операционной системе изолирован от других: он обладает собственным виртуальным адресным пространством, файловыми дескрипторами, переменными окружения и другими ресурсами. При этом изоляция обеспечивает стабильность: сбой одного процесса не влияет на другие.

Поток — это «нить» внутри процесса, исполняющая некоторый код. Все потоки одного процесса разделяют его память, файловые дескрипторы и другие ресурсы. Это делает обмен данными между потоками быстрым, но вместе с тем повышает риск конфликтов при одновременном доступе к одной и той же переменной.

• Виды очередей (swift)

A. Последовательные очереди (Serial Queues)  
· Выполняют задачи строго последовательно, в порядке их добавления  
· Каждая очередь использует один поток  
· Гарантируют порядок выполнения  
· Полезны для синхронизации доступа к общим ресурсам

B. Параллельные очереди (Concurrent Queues)  
· Могут выполнять несколько задач одновременно  
· Используют несколько потоков из системного пула  
· Порядок завершения задач не гарантирован  
· Подходят для независимых задач

C. Системные очереди:  
· Main Queue - специальная последовательная очередь, привязанная к главному потоку приложения  
· Global Queues - четыре предопределенные параллельные очереди с разными приоритетами

!!!Очередь не является потоком!!!

Очередь — это абстрактная структура данных, которая управляет выполнением задач по принципу FIFO (First-In, First-Out) — "первым пришёл, первым ушёл". В контексте многопоточности Swift (GCD) очередь — это механизм, который планирует выполнение задач, а система управляет потоками для их выполнения.

• Grand Central Dispatch (GCD) 

GCD использует абстракцию очередей (queues) вместо прямого управления потоками. Система сама управляет пулом потоков, выделяя и переиспользуя их по мере необходимости.

• Quality of Service (QoS)

Quality of Service (QoS) - это система приоритизации задач в GCD и OperationQueue, которая указывает системе, насколько срочно нужно выполнить работу. Это не про скорость выполнения, а про важность задачи относительно других.

Приоритеты выполнения:  
· .userInteractive - высший приоритет, для анимаций и мгновенного отклика UI  
· .userInitiated - задачи, инициированные пользователем, требующие быстрого результата  
· .default - стандартный приоритет (между .userInitiated и .utility)  
· .utility - длительные задачи, где пользователь ожидает результат  
· .background - низший приоритет, для задач, незаметных для пользователя  
· .unspecified - отсутствие явного приоритета

• Async/Await 

· async — обозначение асинхронной функции  
· await — точка приостановки выполнения  
· Task — единица асинхронной работы

• Виды потоков ios

1Поток (Thread) — это наименьшая единица выполнения в процессе. Каждый процесс имеет хотя бы один основной поток и может создавать дополнительные потоки для параллельного выполнения кода.

2Поток (Main Thread) — это первый и основной поток, создаваемый операционной системой при запуске любого приложения. Это единственный поток, который может взаимодействовать с пользовательским интерфейсом.

3Background Thread (Фоновый поток) — это любой поток, который не является главным потоком приложения. Все потоки, кроме основного, считаются фоновыми 
потоками.

• Operation и OperationQueue

Operation и OperationQueue - Объектно-ориентированная абстракция.Более высокоуровневая абстракция над GCD, где задачи инкапсулируются в объекты Operation, а управление осуществляется через OperationQueue.

Особенности:

· Operation - абстрактный класс, представляющий единицу работы  
· BlockOperation - реализация Operation для выполнения блока кода  
· OperationQueue - управляет выполнением операций, используя GCD под капотом

• Проблемы многопоточности

A. Состояние гонки (Race Condition)

Неопределенный результат из-за неконтролируемого порядка доступа потоков к общим данным.

B. Взаимная блокировка (Deadlock)

Два или более потоков блокируют друг друга, ожидая освобождения ресурсов.

C. Голодание (Starvation)

Поток не получает доступа к ресурсам из-за более высоких приоритетов других потоков.

D. Инверсия приоритетов

Низкоприоритетный поток удерживает ресурс, нужный высокоприоритетному потоку.

• В Заключение

Многопоточность в Swift управляет высокоуровневыми абстракциями, которые обеспечивают безопасность и производительность. Современный подход через async/await и акторы позволяет писать безопасный многопоточный код с минимальным риском ошибок, сохраняя при этом высокую производительность за счет эффективного использования системных ресурсов
